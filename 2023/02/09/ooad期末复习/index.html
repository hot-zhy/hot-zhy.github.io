单实例模式SingleTon

单实例模式的key points

步骤

  1. 定义一个公有的成员函数，创建并供客户获取该单实例
  2. 定义一个私有的构造函数
  3. 定义私有的静态成员变量

单实例模式的类图

锁住某类的实例化功能，客户类只能使用某类自身实例化的唯一实例

summary

要点

  • 只能有一个实例
  • 这个实例能够方便地被所有客户访问

解决方法

  • 定义私有的静态成员变量，保存单实例的引用
  • 定义公有的getter函数
  • 该类自己负责第一次使用时实例化对象
  • 定义私有的构造函数
  • 客户对象只能通过getter函数获得该单例

Scenario

You have an existing software system that you need to work a new vendor library , but the new vendor designed their interfaces differently than the lasted vendor

What to do?

Write a class that adapts the new vendor interface into the one you’re expecting.

外观Facade模式

现有系统功能强大复杂，开发新系统需要用到其中的一部分，但又要增加新的一部分

外观模式的关键点(key features)

外观模式的动机

  • 最小化通信minimize communication
  • 最小化依赖minimize dependencies

这里有一个前台

外观模式和适配器模式的比较

  • 外观定义了新的接口，适配器使用旧的接口
  • 适配器使得两个不一致的接口协同工作，而不是定义一个新的
  • 外观模式的本意是产生一个轻便的接口，适配器的本意是把现有接口转换一下
  • 一个外观接口可能包装了多个现有系统的对象，也可能增加了一些新的功能，而适配器只是包装一个对象
  • 多数情况下，外观是单实例的
  • GRASP原则的外观控制器，就是外观模式

外观模式的适用性

  • 简化复杂系统的使用
  • 在使用复杂系统的某个功能时，可以加入新功能
  • 为客户软件提供一个接口，隐藏了现有系统的细节
  • 使客户与现有系统之间减少耦合，增加独立性，可移植性
  • 节约成本
    • 开发新的外观类的成本，要比每个人都去熟悉现有系统，了解如何使用现有系统的功能要节省许多

观察者模式

当有许多不同的客户都对同一数据源感兴趣，对相同的数据有不同的处理方式，该如何解决？

什么是观察者模式？

定义对象之间的一对多依赖关系，当一个对象改变状态时，所有依赖于它的对象都会自动获得感知

观察者模式案例

观察者模式又称发布-订阅模式(publish /Subscribe)，源-监听器(Source/Listener)模式，模型-视图模式(Model/View)

  • Publisher==Subject 主题

  • Subscribers==Observers 观察者

    观察者已经订阅了主题数据，当数据发生变化之后将会收到更新数据或更新通知

观察者模式信息的推拉模式

推模式

当通知消息来的时候，把所有相关消息都通过参数的形式“推给”观察者

  1. 优点
    • 所有信息通过参数传递，直接，简单，观察者可以马上处理
    • 观察者和被观察者没有一点联系，两者之间几乎没有耦合
  2. 缺点
    • 所有信息强迫推给观察者，不管有用与否
    • 如果想添加一个参数，那就需要修改所有观察者的接口函数

拉模式

当通知消息来的时候，通知的函数不带任何相关的信息，而是要观察者主动去被主题对象那里去“拉”信息

  1. 优点
    • 可以主动去取自己感兴趣的信息
    • 如要添加一个参数，无需修改观察者
  2. 缺点
    • 观察者与被观察者有一定的联系

如何应用观察者模式 check list

  • 主题对象只与观察者基类有耦合
  • 客户配置观察者的数量和类型
  • observers首先要知道Subject，然后把自己注册到Subject中
  • Subject保存所有注册过的Observer，当状态发生变化时，广播给所有的注册观察者
  • Subject可以采用push或pull的模式，与Observer交流信息

策略模式

OO原则：Open-Closed Principle（OCP）原则

  • open for extension-the behavior can be extended to meet new requirements
  • closed for Modification-the source code of the module is not allowed to change

设计原则

软件行业公理

在软件项目中，唯一不变的事情就是变化

解决办法

拥抱（embrace）变化，让变化成为设计的一部分

标识变化点，把它们和系统其余部分隔离开

利用接口的强大功能

能用组合的地方不要用继承
