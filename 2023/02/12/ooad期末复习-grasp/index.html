OOAD期末复习——GRASP

GRASP-General Responsibility Assignment Software Patterns

通用职责分配软件模式

Creator—创建者

Name：

Creator

Problem

由谁来创建某个类的新实例（对象）

Solution

我们分配给B创建A的实例的职责，当且仅当下面条件中有一条成立：

  1. B contains or compositely aggregates A
  2. B records A
  3. B closely uses A
  4. B has the initializing data for A that will be passed to A when it is created (B is an expert with respect to A)

When not to use

  • When wants to reuse existing instances for performance purposes (caching)
  • 创建大批的对象——应该使用工厂模式
  • Delegate responsibility further down (委托)

Benefits

  • Existing associations means created class is in any case visible to creator
  • High cohesion
  • does not increase coupling

Information Expert—信息专家

problem

What is a general principle of assigning responsibility to objects ?

Solution

Assign responsibility to the class that has the information necessary to fulfill responsibility

步骤

  1. Clearly state the responsibility
  2. Look in Design model for relevant classes
  3. Else look in Domain model and create design classes

优点

  • 封装性
    • 对象充分利用自身的信息
    • 支持低耦合
  • 系统行为分布到不同的类
    • 支持高内聚

Low coupling低耦合

耦合的定义

  • 耦合：一个元素与其他元素的联接，感知以及依赖程度的度量
  • 比较
    • 内聚：cohesion：模块内的操作之间联系紧密的程度
    • 耦合：coupling：两个子模块之间联系的程度
  • 高耦合带来的问题：
    • 牵一发而动全身
      • A依赖B，A与B之间有耦合，一旦B变化了，A就会受到影响
    • 元素很难重用

Problem

How to support low dependency,low change impact and increase reuse?

Solution

Assign responsibility so coupling remains low

  • use this principle to evaluate alternatives
  • All other things being equal prefer the low coupling solution

Information expert support low coupling

discuss

低耦合是所有设计决策时自然而然要考虑的原则

低耦合是一种评估原则，是设计师用来对设计方案进行评价的一种指标

低耦合支持类的设计相对独立，减少了变化带来的相互影响

低耦合与其他的原则必须综合考虑

在继承关系中，子类与父类的耦合十分紧密，因此，能用组合的地方不要用继承

低耦合不具备可操作性

极端情况下，类之间没有耦合，不希望这种情况出现

类之间存在适度的耦合是正常的，必须的

Controller—控制器

problem

UI应该把捕捉到的系统操作发给领域层的哪个对象呢？

What first object beyond the UI layer receives and co-ordinates a system operation?

solution

Assign responsibility to receive the system operations by an object which

  • representing system /subsystem /device or
  • handling just this use case

Assign responsivity to a class representing one of the following choices:

  • Facade controller

  • Use case or session controller

什么时候用外观控制器，什么时候用会话控制器呢？

控制器的优点

  • 容易适应ui层的变化
  • 领域层代码易于重用
  • 有助于保证应用所需要的操作顺序
  • 可以对系统的状态进行推理（ui层不保存系统状态）

bloated controllers——臃肿的控制器

high Cohesion—高内聚

Problem

How to keep objects focused ,understandable and manageable ,and as a side effect support low coupling

solution

Assign responsibility so cohesion remains high

usage

  • used as a evaluation tool
  • 更多的是一种理念，没有具体的可操作原则

cohesion defined

  • 衡量概念之间相关度的两个指标
    • 内聚：模块内元素之间联系紧密的程度，比如，一个类内部的操作之间
    • 耦合：两个模块之间联系的强度
  • 内聚的最佳实践
    • 一个对象完成的功能不是太多
    • 这些功能都是同一类别的
    • 例如，教授的主要任务就是教学，研究员的主要任务就是科研

类低内聚

症状

  • 做了太多相互无关的动作
  • 做了太多工作

原因

  • 大粒度的抽象
  • 做了太多本应该委托给其他类去做的工作

问题

  • 难以理解
  • 难以重用
  • 难以维护
  • 没有稳定的时刻，总是在修改(通常都会高耦合)

summary

高内聚的类

  • 有较少数量的工作，操作的性质基本一致，不会做太多的事情
  • 如果同类别的工作太多，则会定义新的类分担任务，相互间合作

高内聚的类有许多优点:

  • 易于理解
  • 易于维护
  • 易于重用

高内聚也是一种评估性原则，用于评估所有的设计决策是否合适

Polymorphism—多态

problem

如何处理依据类型不同而有不同行为的一类需求

solution

使用多态操作为依据类型变化的行为进行职责分配

corollary

  • 不要去测试对象的类型或条件逻辑，并以此选择相应的行为
  • 即，不要使用条件逻辑，而是为不同的类定义相同名字的方法
  • 不同的类实现了相同的接口，或者由一个共同的父类

Pure Fabrication—纯虚构

problem

依据一些原则获得的解决方案不合适的情况下，既不想违反低耦合，高内聚，也不想违反其他的原则，如何把职责分配给对象？

solution

把高度内聚的职责分配给虚构出来的一个类，这个类在领域模型中没有对应的概念

例子

纯虚构原则

  • 拎出来的系统可重用
  • 符合高内聚低耦合
  • 多数情况下是按照功能类定义新的类，所以是一种以功能为中心的对象
  • 如果功能的相关性比较高的情况下可满足高内聚

Indirection—间接

problem

把职责分配到哪里可以避免两个或多个对象之间的直接耦合？如何解耦对象以保证较高的可重用性？

solution

把职责分配给一个中介对象，隔离对象与其他构件或者服务，使他们不产生直接耦合

protected variations—隔离变化

problem

如何设计对象、系统和子系统，使得这些成分里面的变化因素，不稳定因素不会对系统造成意想不到的影响？

solution

标识出能够预计的变化点或不稳定点，职责分配的时候创建一个稳定的接口，把他们与系统的其余部分隔离开来

需要注意的两种可能的变化点

  • 变化点：在当前系统或者需求中已经存在了
  • 演化点：推测的类型变化可能发生在今后，但在当前的需求中

OCP—开-闭原则

设计变坏的前兆

  • 僵硬性
    • 难以更改代码
    • 从管理角度，拒绝任何的变化成为一种制度
  • 易碎性
    • 即使是小小的改动也是会导致级联型的后果
    • 代码在意想不到的地方终止
  • 固定性
    • 代码纠缠在一起根本不可能重用
  • 粘滞性
    • 宁愿重新编写也不愿意修改

设计变坏的原因

  • 需求不断在变化
  • 设计的问题：依赖管理失衡

开-闭原则

  • 软件系统应当允许功能扩展（即开放性）open for extension
  • 但是不允许修改原有的代码（即关闭性）closed for modification

ocp的启发

  • 定义所有的对象、数据为私有的
  • 不要使用全局变量

summary

  • ocp解决软件的僵硬性和易碎性
  • 我们应当尝试设计永远不需要修改的模块
  • 系统行为的扩展只需要增加新的代码，不能修改已有的代码
  • 模块不允许修改已有的代码，而这种修改会影响客户端

能用组合的地方不要用继承

关于类的继承

  • 可以使用继承，但设计师必须确信使用继承是很好的解决方案时才使用

组合的特点

对象组合

  • 没有打破封装性
  1. 对象组合是一种动态/运行时绑定
  2. 在运行时切换对象引用，就可以改变行为
  3. 整体与部分之间只有接口边界关联，耦合较低
    • 对私有数据不存在“窥视与拨弄”
    • 不存在大量无用代码一类的负担
  4. 各部分的职责明确
    • 每个对象清晰地集中在少量的任务上
    • 只要阅读了接口，就可以了解系统
    • 容易独立测试
      • 使用得当，将导致更可靠的设计
    • 每个类依然苗条（高内聚）
      • 避免巨无霸这种反模式的现象，即一个类似乎完成了几乎全部的功能
    • 重用的几率大增

操作契约

什么是操作契约

  • a way to describe use cases in more detail
  • part of the Use Case Model

格式

  • operation：操作名称与参数
  • cross references：操作所属的用例
  • preconditions：前置条件
  • postconditions：当操作完成后，领域模型里面的对象应该保证的状态
    • 后置条件描述了领域对象状态的变化
      1. 是否创建、删除了对象
      2. 对象间的关系是否发生变化
      3. 对象的属性是否发生变化

操作契约比较细节地、精确地描述了一个操作应当承担的职责

在定义契约的过程中，经常触发对领域模型的修改

  • 记录新的概念类、属性
  • 领域模型中新的关系

系统顺序图

系统顺序图与顺序图的区别

  • SSD：to emphasize to treat system as black boxes
  • SD: be used to illustrate the design of interacting software objects to fulfill work

面向对象分析方法—名词法

analysis phase

  • 系统规格说明，也称为逻辑结构，是开发人员眼中的系统
  • 大的、复杂的系统的开发，有两种主要的分析方法：
    • 面向功能的分析——把系统划分为模块
    • 面向对象分析——模块功能的实现

面向对象分析的步骤

  1. 识别对象
  2. 组织对象
  3. 定义对象之间的关系
  4. 定义对象的操作 （设计阶段）
  5. 定义对象内部细节

三种面向对象分析的方法

  • 名词法——
  • 分析模型——boundaries entities control
  • CRC——类、职责、协作

目标：识别对象、定义对象之间的关系

名词法

  • 重用或修改已有的模型
  • 借助行业、公司内部法的“概念类列表” Concept Category list Check list
  • 在需求描述中查询名词或名词短语

面向对象分析与结构化分析之间的最大差异是：

前者根据对象划分系统，而后者根据功能

方法

  • 在问题领域的文本描述中，标识出名词和名词短语，把他们作为候选的概念类或属性
  • 对发现的名词进行分析，辨别是合适的概念类吗，概念可以合并吗
  • 定义概念类之间的关系
  • 定义概念类的属性

分析模型法

  • 实体类 entity

    identifying entity objects

    an entity object models information that shows the state of a system

  • 边界类 boundary

    identitying interface objects

    a boundary / interface object models input and output operations that process them

  • 控制类 control

    identifying control objects

    a control object models functionality / operations regarding to validate and decide whether to process and pass information from the interface object to the entity object or the way around
